# Tech Debt Register

> Реестр архитектурного технического долга. Ведёт [Technical Architect](../docs/roles.md#technical-architect-technology-architect), пересматривается каждый релиз / квартал.

## Принципы ведения

- Каждый элемент долга — осознанный компромисс, зафиксированный явно
- Долг без владельца = долг, который никто не устранит → всегда назначать owner
- Долг без оценки стоимости = долг, который невозможно приоритизировать
- Пересмотр: минимум раз в квартал или при каждом релизе

## Классификация

| Категория | Описание | Примеры |
|-----------|---------|---------|
| **Security** | Компромиссы в безопасности | Отсутствие encryption at rest, устаревшие зависимости |
| **Scalability** | Ограничения масштабирования | Stateful sessions, single-instance DB |
| **Reliability** | Риски доступности | Отсутствие failover, нет circuit breaker |
| **Performance** | Известные узкие места | N+1 запросы, отсутствие кэширования |
| **Maintainability** | Сложность сопровождения | God classes, circular dependencies, low coverage |
| **Cost** | Неоптимальные расходы | Overprovisioned ресурсы, отсутствие autoscaling |
| **Compliance** | Несоответствие стандартам | Устаревшие API-версии, нарушение naming conventions |

## Severity

| Уровень | Описание | Действие |
|---------|---------|----------|
| 🔴 **Critical** | Блокирует масштабирование / создаёт security risk / потенциальный инцидент | Планировать устранение в ближайший спринт |
| 🟡 **High** | Замедляет разработку / увеличивает cost / снижает reliability | Планировать в ближайший квартал |
| 🟢 **Medium** | Неудобство, не критичное | Устранять при возможности |
| ⚪ **Low** | Косметическое / minor | Backlog, по желанию |

## Статусы

| Статус | Описание |
|--------|---------|
| 📝 Зафиксировано | Долг обнаружен, оценён, но не запланирован |
| 📋 Запланировано | Включён в план устранения |
| 🔧 В работе | Устраняется в текущем спринте |
| ✅ Устранён | Долг закрыт (перенести в архив) |
| ❌ Принят | Осознанно оставлен (зафиксировать причину в ADR) |

## Процесс управления

### 1. Фиксация
При обнаружении долга (code review, architecture review, инцидент) — добавить запись в реестр с категорией и severity.

### 2. Оценка
SA/TA оценивают стоимость устранения и риск при неустранении. Связывают с ADR, если долг — результат осознанного решения.

### 3. Приоритизация
На planning совместно с Product Owner:
- 🔴 Critical → в ближайший спринт
- 🟡 High → в ближайший квартал
- 🟢 Medium/Low → backlog

Рекомендация: выделять 15–20% capacity на устранение техдолга.

### 4. Исполнение
Устранение долга проходит стандартный процесс: PR → code review → CI gates.

### 5. Пересмотр
Каждый квартал (или каждый релиз):
- Актуализировать severity (контекст мог измениться)
- Проверить, не появился ли новый критичный долг
- Архивировать устранённые элементы
- Обновить метрики (см. ниже)

## Метрики техдолга

| Метрика | Формула | Target |
|---------|---------|--------|
| Общее количество | Count(all items) | Тренд на снижение |
| Critical items | Count(severity = Critical) | 0 |
| Средний возраст | Avg(today - дата обнаружения) | < 90 дней |
| Velocity устранения | Count(устранённых за квартал) | ≥ Count(новых за квартал) |
| % capacity на техдолг | Время на техдолг / общее время | 15–20% |
