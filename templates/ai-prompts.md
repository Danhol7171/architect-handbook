# AI Prompt Kit для архитекторов

> Набор промптов для повышения эффективности архитектурной работы. Используйте с чат-ассистентом (LLM) или терминальным AI-агентом.

## 1. Анализ требований и формирование вариантов

```
Ты — опытный Solution Architect. Проанализируй следующие требования и предложи 2-3 варианта архитектурного решения.

Требования:
[вставить описание задачи, NFR, ограничения]

Для каждого варианта укажи:
- Краткое описание подхода
- Ключевые технологии
- Плюсы и минусы
- Примерная оценка сложности (низкая/средняя/высокая)
- Риски

В конце дай рекомендацию с обоснованием. Ответ оформи как черновик ADR.
```

## 2. Генерация C4-диаграмм

```
Сгенерируй C4 Context diagram в формате Mermaid для следующей системы:

Система: [название]
Пользователи: [перечислить]
Внешние системы: [перечислить]
Ключевые потоки данных: [описать]

Используй формат graph TD. Для стилей используй:
- Целевая система: fill:#3B2066,color:#fff
- Внешние системы: без заливки
- Подписи: используй <br> для переноса строк в узлах
```

```
На основе C4 Context выше, сгенерируй C4 Container diagram.
Детализируй внутреннюю структуру системы:
- Фронтенд (технология)
- Backend-сервисы (технология для каждого)
- Базы данных
- Очереди/брокеры сообщений
- Внешние API/интеграции

Формат: Mermaid graph TD.
```

## 3. Ревью архитектуры

```
Ты — независимый архитектурный ревьюер. Проведи ревью следующего архитектурного решения по критериям:

1. Reliability — отказоустойчивость, SLA, DR
2. Security — аутентификация, шифрование, attack surface
3. Performance — узкие места, масштабируемость
4. Cost — оптимальность использования ресурсов
5. Maintainability — сложность, связанность, техдолг

Архитектура:
[вставить описание или C4-диаграмму]

Для каждого критерия:
- Оценка (хорошо / есть замечания / проблема)
- Конкретные замечания
- Рекомендации по улучшению

В конце — топ-3 приоритетных действия.
```

## 4. Reverse engineering кодовой базы

```
Проанализируй кодовую базу и создай архитектурное описание:

1. C4 Context — какие внешние системы/пользователи взаимодействуют
2. C4 Container — какие сервисы/приложения/БД есть
3. Ключевые зависимости между модулями
4. Используемые паттерны (MVC, CQRS, event-driven и т.д.)
5. Точки интеграции с внешними системами
6. Потенциальные проблемы и техдолг

Результат оформи в Markdown с Mermaid-диаграммами.
```

## 5. Генерация NFR checklist

```
Сгенерируй NFR checklist для проекта со следующим контекстом:

Домен: [e-commerce / fintech / healthcare / ...]
Пользователи: [количество, география]
Критичность: [mission-critical / business-critical / internal tool]
Регуляторные требования: [152-ФЗ / PCI DSS / HIPAA / нет]
Бюджет инфраструктуры: [примерный]

Для каждой категории (Scalability, Performance, Availability, Security, Recoverability, Maintainability, Operability, Cost) предложи:
- Конкретные метрики с целевыми значениями
- Приоритет (Must/Should/Could)
- Способ проверки
```

## 6. Threat Modeling

```
Проведи threat modeling по методологии STRIDE для следующей системы:

[вставить C4 Container diagram или описание]

Для каждого компонента и потока данных определи:
- Возможные угрозы (Spoofing, Tampering, Repudiation, Information Disclosure, Denial of Service, Elevation of Privilege)
- Вероятность и влияние
- Рекомендуемые контрмеры

Результат оформи как таблицу. Отдельно выдели топ-5 критичных угроз.
```

## 7. Генерация OpenAPI спецификации

```
Сгенерируй OpenAPI 3.1 спецификацию для следующего API:

Ресурсы: [перечислить]
Операции: [CRUD + специфичные]
Аутентификация: [Bearer JWT / API Key / OAuth 2.0]
Формат ответов: JSON

Требования:
- operationId для каждой операции
- Описания на русском
- Примеры request/response
- Стандартные коды ошибок (400, 401, 403, 404, 500)
- Пагинация для list-операций
```

## CLAUDE.md шаблон для архитектурного проекта

Файл `CLAUDE.md` размещается в корне репозитория и задаёт контекст для AI-агента:

```markdown
# Архитектурный контекст проекта

## Система
[Название] — [краткое описание назначения]

## Архитектура
- Стиль: [микросервисы / монолит / modular monolith]
- Основные языки: [Java, Go, TypeScript, ...]
- Инфраструктура: [Kubernetes / VM / serverless], [облако]
- БД: [PostgreSQL, Redis, ClickHouse, ...]
- Брокер: [RabbitMQ / Kafka / нет]

## Ключевые файлы
- `architecture/README.md` — Solution Architecture Document
- `decisions/` — Architecture Decision Records
- `quality/nfr-checklist.md` — нефункциональные требования
- `api/openapi.yaml` — API спецификация

## Правила
- Все архитектурные решения фиксируем в ADR (`decisions/NNNN-*.md`)
- Диаграммы — только Mermaid (не PlantUML)
- API — contract-first (OpenAPI 3.1)
- Все сервисы должны экспортировать метрики (OpenTelemetry)
- Код ревью обязателен для merge

## NFR
- Латентность API: p95 < [X] мс
- Доступность: [X]%
- Бюджет инфраструктуры: < [X] ₽/мес
```
