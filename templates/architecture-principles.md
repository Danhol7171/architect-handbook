# Architecture Principles: [Название проекта]

> Пример. Принципы оформляются как ADR и фиксируются в начале проекта.

## Принципы

### ADR-0001: Managed services вместо self-hosted

**Статус:** accepted

Используем managed-версии сервисов (БД, очереди, мониторинг) вместо самостоятельного развёртывания. Снижаем операционную нагрузку на команду, принимаем риск vendor lock-in.

### ADR-0002: API-first

**Статус:** accepted

Все межсервисные взаимодействия начинаются с проектирования API (OpenAPI/AsyncAPI). Контракт согласуется до начала реализации. Это позволяет параллельную разработку и раннее обнаружение проблем интеграции.

### ADR-0003: Docs-as-code

**Статус:** accepted

Вся архитектурная документация хранится в Git рядом с кодом. Диаграммы в Mermaid. Ревью через merge request. Это обеспечивает версионирование, трассируемость и актуальность документации.

### ADR-0004: Наблюдаемость по умолчанию

**Статус:** accepted

Каждый сервис обязан экспортировать метрики, логи и трассировки (OpenTelemetry). Наблюдаемость — не опция, а требование к архитектуре. Без неё сервис не допускается в production.

### ADR-0005: Минимизация связанности

**Статус:** accepted

Сервисы взаимодействуют через асинхронные события (event-driven) где возможно. Синхронные вызовы — только для real-time операций. Это повышает устойчивость и позволяет независимый деплой.
