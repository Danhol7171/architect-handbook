# Integration Review Checklist

> Чеклист для ревью интеграционного дизайна. Проходите перед началом реализации каждой интеграции.

## Контракт

- [ ] API контракт описан (OpenAPI для sync, AsyncAPI для async)
- [ ] Формат данных согласован обеими сторонами
- [ ] Версионирование API определено (URL path / header)
- [ ] Breaking changes policy согласована

## Надёжность

- [ ] Определена стратегия retry (количество, backoff, timeout)
- [ ] Определено поведение при недоступности внешней системы (fallback, circuit breaker)
- [ ] Настроен Dead Letter Queue (DLQ) для необработанных сообщений
- [ ] Обеспечена идемпотентность (повторная обработка не создаёт дубликатов)
- [ ] Определены таймауты для всех вызовов

## Данные

- [ ] Описан маппинг данных (source → target)
- [ ] Определена стратегия валидации входных данных
- [ ] Продуманы граничные случаи (пустые значения, максимальные размеры, спецсимволы)
- [ ] Определён максимальный размер payload

## Безопасность

- [ ] Аутентификация настроена (API key / OAuth / mTLS)
- [ ] Данные передаются по защищённому каналу (TLS)
- [ ] PII/чувствительные данные маскируются в логах
- [ ] Настроен rate limiting (если применимо)

## Наблюдаемость

- [ ] Логируются все вызовы (request/response, без PII)
- [ ] Настроены метрики (количество вызовов, латентность, ошибки)
- [ ] Настроены алерты на аномалии (рост ошибок, увеличение латентности)
- [ ] Есть correlation ID для сквозной трассировки

## Производительность

- [ ] Определён ожидаемый объём (RPS / сообщений в час)
- [ ] Проведена оценка влияния на latency основного сценария
- [ ] Batch-обработка используется где возможно (вместо поштучной)
- [ ] Определён SLA интеграции (максимальная задержка доставки)

## Эксплуатация

- [ ] Описана процедура переподключения при сбое
- [ ] Есть runbook для типовых проблем
- [ ] Определён ответственный за интеграцию (ownership)
- [ ] Есть способ отключить интеграцию без деплоя (feature flag / config)
