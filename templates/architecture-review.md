# Architecture Review Checklists

## 1. Design Review (до реализации)

Проводится перед началом реализации. Цель — убедиться, что архитектура продумана и обоснована.

### Общее

- [ ] Есть C4 Context diagram (L1)
- [ ] Есть C4 Container diagram (L2) для ключевых систем
- [ ] Архитектурные решения зафиксированы в ADR
- [ ] Определены границы системы (что входит / не входит)
- [ ] Стейкхолдеры согласовали scope

### Reliability

- [ ] Определены SLA/SLO (доступность, латентность, RTO, RPO)
- [ ] Описана стратегия отказоустойчивости (retry, circuit breaker, fallback)
- [ ] Продуман disaster recovery (DR)
- [ ] Нет single points of failure для критичных компонентов
- [ ] Описана стратегия деплоя (blue-green, canary, rolling)

### Security

- [ ] Определена модель аутентификации и авторизации
- [ ] Продумана защита данных at rest и in transit
- [ ] Проведён threat modeling (или запланирован)
- [ ] Нет хардкода секретов в коде/конфигурации
- [ ] Соответствие регуляторным требованиям (152-ФЗ, PCI DSS и т.д.)
- [ ] Описана стратегия audit logging

### Performance

- [ ] Определены целевые метрики (p50/p95/p99, RPS, concurrent users)
- [ ] Продуманы узкие места (hot paths) и стратегия кэширования
- [ ] Описана стратегия масштабирования (horizontal/vertical)
- [ ] Нет N+1 паттернов в дизайне интеграций

### Cost

- [ ] Есть cost model (ежемесячные затраты на инфраструктуру)
- [ ] Продуманы оптимизации (reserved instances, auto-scaling, CDN)
- [ ] Определены алерты на превышение бюджета

### Data

- [ ] Определена модель данных (ER-диаграмма или описание)
- [ ] Продумана стратегия миграции данных (если есть legacy)
- [ ] Описана стратегия бэкапов и восстановления
- [ ] Определены требования к консистентности (strong/eventual)

### Integration

- [ ] Описаны все интеграционные потоки
- [ ] API контракты определены (OpenAPI/AsyncAPI)
- [ ] Продуман error handling для каждой интеграции
- [ ] Определены SLA интеграций

### Операции

- [ ] Описаны требования к наблюдаемости (metrics, logs, traces)
- [ ] Продуманы runbooks для типовых инцидентов
- [ ] Определены on-call процессы

## 2. Architecture Assessment (аудит существующей системы)

Проводится для оценки текущего состояния архитектуры. Цель — выявить проблемы, риски и точки улучшения.

### Текущее состояние

- [ ] Восстановлен C4 Context (as-is)
- [ ] Восстановлен C4 Container (as-is)
- [ ] Идентифицированы ключевые технологии и версии
- [ ] Определены владельцы компонентов (ownership)
- [ ] Собрана информация о текущих SLA/SLO

### Reliability

- [ ] Проверена фактическая доступность (uptime за последние 3-6 мес)
- [ ] Выявлены single points of failure
- [ ] Проверена работоспособность DR (когда последний раз тестировали?)
- [ ] Проверены бэкапы (восстановление протестировано?)
- [ ] Оценена устойчивость к нагрузке (есть ли результаты нагрузочного тестирования?)

### Security

- [ ] Проведено сканирование зависимостей (CVE)
- [ ] Проверена конфигурация TLS/шифрования
- [ ] Проверены права доступа (принцип наименьших привилегий)
- [ ] Оценена полнота audit log
- [ ] Проверено соответствие регуляторным требованиям

### Performance

- [ ] Собраны метрики латентности (p50/p95/p99)
- [ ] Идентифицированы bottlenecks
- [ ] Проверена утилизация ресурсов (CPU, память, диск, сеть)
- [ ] Оценена эффективность кэширования

### Качество кода и техдолг

- [ ] Проведён анализ качества кода (сложность, покрытие, дупликация)
- [ ] Идентифицированы hotspots (часто меняемые файлы с высокой сложностью)
- [ ] Оценён уровень техдолга (критический / управляемый / низкий)
- [ ] Проверены зависимости (устаревшие, с CVE, без поддержки)

### Операции

- [ ] Проверена полнота мониторинга и алертинга
- [ ] Оценено время деплоя и частота релизов (DORA метрики)
- [ ] Проверена документация (актуальна? достаточна?)
- [ ] Оценён bus factor

### Cost

- [ ] Собраны текущие затраты на инфраструктуру
- [ ] Выявлены неиспользуемые ресурсы
- [ ] Оценена стоимость на транзакцию/пользователя
