# Инструменты и нотации

## Два слоя архитектурного инструментария

Разделяем инструменты на два слоя, потому что задачи разные:

1. **Delivery-архитектура** (solution/technical) - живёт рядом с кодом, обновляется в проекте
2. **Enterprise-слой** (портфели, capability, roadmaps) - ведётся в EAM-платформе, питается фактами из delivery

---

## Профили инструментария

Не на каждом проекте нужен полный стек. Профиль определяется [уровнем rigor](core-standard.md#уровни-rigor).

| Инструмент | Lite (L0–L1) | Standard (L2) | Enterprise (L3) |
|-----------|:---:|:---:|:---:|
| **Диаграммы:** Mermaid | ✅ | ✅ | ✅ |
| **Диаграммы:** Structurizr DSL | — | ✅ | ✅ |
| **Диаграммы:** ArchiMate / EAM | — | — | ✅ |
| **Документация:** Markdown + Git | ✅ | ✅ | ✅ |
| **Документация:** arc42 template | — | ✅ | ✅ |
| **Документация:** Backstage TechDocs | — | — | ✅ |
| **Решения:** ADR в Git | ✅ | ✅ | ✅ |
| **Решения:** Log4brains (веб-UI) | — | ✅* | ✅ |
| **API:** OpenAPI/AsyncAPI YAML | — | ✅ | ✅ |
| **API:** Spectral lint в CI | — | ✅ | ✅ |
| **Policy:** OPA/Conftest | — | ✅ | ✅ |
| **CI:** Markdown lint + link check | — | ✅ | ✅ |
| **CI:** Fitness functions | — | ✅ | ✅ |
| **CI:** Drift detection | — | — | ✅ |
| **Каталог:** Backstage Catalog | — | — | ✅ |
| **AI:** LLM-ассистент | ✅* | ✅ | ✅ |
| **AI:** MCP для корпоративных данных | — | ✅* | ✅ |

\* Опционально.

**Lite** — минимальный набор для аудитов и пресейла. Mermaid-диаграммы в Markdown, ADR в Git, AI-ассистент по желанию. Развернуть за 5 минут.

**Standard** — основной набор для delivery-проектов. Добавляет model-as-code (Structurizr), CI-автоматизацию (lint, policy-as-code, fitness functions), contract-first API design. Развернуть из [Starter Kit](artifacts.md#starter-kit) за 15 минут.

**Enterprise** — полный стек для программ и трансформаций. Добавляет ArchiMate/EAM, Backstage как каталог и портал документации, drift detection. Требует отдельной настройки инфраструктуры.

## Constrained Mode (без Git для документации)

Методика ориентирована на Git/CI-first подход, но на enterprise-клиентах это не всегда возможно: закрытый контур, политики клиента, запрет на внешние инструменты. В таких случаях используется constrained mode.

**Минимальный эквивалент:**
- ADR ведутся в Confluence/SharePoint с review-процессом (комментарии = аналог PR review)
- C4-диаграммы -- в любом diagram tool (draw.io, Visio, Mermaid Live Editor) с экспортом в wiki
- NFR Checklist, Risk Register -- в виде таблиц в wiki или Excel с версионированием
- Шаблоны из Starter Kit копируются вручную

**Quality gates без CI:**
- Ручной чеклист перед каждой сдачей (вместо автоматических gates)
- Периодический review артефактов (раз в 2 недели вместо automated lint)
- Governance через регулярные встречи (вместо PR-based async review)

**Правило:** при появлении возможности использовать Git -- мигрировать артефакты. Constrained mode -- временная мера, не целевое состояние.

## Нотации

### C4 Model - основной стандарт

C4 (Context, Container, Component, Code) - де-факто стандарт для solution/technical архитектуры. Используем по умолчанию на всех проектах. Примеры: [C4 Context](artifacts.md#c4-context-diagram-level-1), [C4 Container](artifacts.md#c4-container-diagram-level-2).

**Обязательные уровни:**
- **Context (L1)** - система в окружении пользователей и внешних систем. Делается всегда
- **Container (L2)** - основные технические блоки (приложения, БД, очереди). Делается всегда

**По необходимости:**
- **Component (L3)** - внутренняя структура контейнера. Для сложных модулей
- **Code (L4)** - обычно не нужен (код и так в репозитории)

### ArchiMate 3.2

Для enterprise-уровня: capability maps, value streams, target operating model. Используем когда клиент работает в EA-языке. На проектном уровне избыточен.

### Классические нотации

UML Component/Package/Sequence Diagrams - используем если клиент требует или команда привыкла. Не навязываем.

## Моделирование

### Diagrams as code - стандартный подход

"Двухскоростной стандарт" для практики:

**Mermaid - по умолчанию (mainstream)**
- Плюсы: встроен в GitHub/GitLab markdown, быстро, нулевой порог входа
- Минусы: ограниченная выразительность для сложных диаграмм
- Когда: быстрые диаграммы, короткие проекты, аудиты, документация в markdown

**Structurizr DSL - для сложных проектов (mainstream/early adopter)**
- Плюсы: единая модель → много диаграмм, контроль через Git, воспроизводимость
- Минусы: нужно обучение DSL, дисциплина, не для всех проектов
- Когда: долгие проекты, сложные системы, когда нужна единая модель архитектуры

**PlantUML - альтернатива (mainstream)**
- Плюсы: зрелый, много типов диаграмм, CI-интеграция
- Минусы: риск "зоопарка" без стандартизации шаблонов
- Когда: если команда уже использует, для UML sequence/activity

**D2 - наблюдаем (early adopter)**
- Лаконичный язык, красивый рендер. Пока слишком мало распространён для стандартизации.

### Коллаборативные инструменты

**Для workshop/discovery/фасилитации:**
- Miro, FigJam, Excalidraw - whiteboard для совместных сессий
- Используются на пресейле, в discovery-фазе, на воркшопах с заказчиком
- Результат потом переносится в diagrams-as-code для долгой жизни

### IcePanel (early adopter)

Визуальная C4-платформа с UI. Полезна когда нужна коллаборация не-технических людей над C4-моделью. Рассматриваем для крупных ландшафтных работ.

### EAM-платформы

Для enterprise-уровня. Выбор зависит от клиента и задачи:

| Платформа | Фокус | Когда рекомендовать |
|-----------|-------|-------------------|
| SAP LeanIX | Application portfolio, technology risk | Оптимизация портфеля, rationalization |
| Ardoq | Data-driven EA, AI-аналитика | Когда нужна интеграция с delivery-данными |
| MEGA HOPEX | EA + GRC, compliance | Зарегулированные отрасли |
| BOC ADOIT | ArchiMate, operating model | Клиенты с зрелым EA на ArchiMate |

**Минимальный набор EA-артефактов** (не надо "вести всё"):
- Capability map
- Application portfolio view
- Target architecture principles
- Roadmap зависимостей

## Документирование

### Docs-as-code - целевой подход

Архитектурная документация живёт в Git рядом с кодом. Это даёт code review, blame, ветки, CI.

**arc42 - шаблон архитектурной документации (mainstream)**
- Прикладная структура без TOGAF-томов
- Хорошо ложится в markdown/AsciiDoc
- Удобен для консалтинга: даёт структуру документа, не перегружает
- [Шаблон Solution Architecture Document](artifacts.md#solution-architecture-document)

**Генераторы документации:**
- **MkDocs (Material)** - для проектной документации в репозитории. Лёгкий, markdown-based
- **Docusaurus** - когда нужен полноценный документационный сайт с версиями
- **Backstage TechDocs** - единая витрина для практики (шаблоны, референсы, гайды). Целевой вариант для масштаба

**Хранение:**
- Git как source of truth для ADR, C4, шаблонов, RFC
- Confluence/вики - как "фронтенд знаний" для не-технических стейкхолдеров. Не единственное хранилище

### ADR-инструменты

**adr-tools (CLI)** - минималистичный путь: ADR в виде markdown-файлов в Git. Рекомендуем как стандарт. См. [практику ADR](practices.md#1-architecture-decision-records-adr) и [шаблон](artifacts.md#architecture-decision-records-adr).

**Log4brains** - если нужен красивый веб-UI для decision log. Генерирует сайт из ADR.

**Формат хранения:**
```
/decisions/
  0001-use-postgresql.md
  0002-api-gateway-pattern.md
  0003-event-driven-for-notifications.md
  README.md  # decision log
```

## AI-инструменты

### Принципы использования AI

AI - это "второе кресло архитектора", не автопилот. Управляемое усиление с обязательным человеческим решением.

### Сценарии применения

**Ежедневная работа (mainstream):**
- Анализ требований и структурирование вариантов решений
- Черновики ADR, документации, коммуникационных материалов
- Рассмотрение trade-offs и альтернатив
- Класс инструментов: чат-ассистент (LLM), веб-интерфейс или десктоп-приложение

**Анализ кодовых баз (быстро растёт):**
- Reverse engineering архитектуры на аудитах
- Карта доменов, модулей, зависимостей из кода
- Навигация по чужому коду
- Класс инструментов: терминальный AI-агент (CLI), AI-ассистент в IDE, AI-поиск по репозиториям

**Управление техдолгом:**
- Инструмент поведенческого анализа кода - hotspots, "где риск и стоимость изменений максимальны"
- Анализатор качества кода - измеримый техдолг, оценка стоимости устранения

**Генерация диаграмм (early adopter → mainstream):**
- Создание C4/Mermaid диаграмм из текстовых описаний
- Работает хорошо только с шаблонами промптов и чёткими входными данными

### MCP (Model Context Protocol)

Стандарт подключения AI-ассистентов к корпоративным системам (репозитории, бизнес-инструменты, dev environments). Ключевой для масштабирования AI без "копипасты секретов".

### AI Prompt Kit

Командные стандарты вместо личных промптов. Готовые [промпты для архитекторов](../templates/ai-prompts.md). Формат:

```
Контекст: [описание системы, ограничения, стек]
Цель: [что нужно получить]
Ограничения: [бюджет, сроки, совместимость, регуляторика]
Критерии качества: [NFR, SLA, compliance]
Формат ответа: [ADR / список вариантов / диаграмма / чеклист]
```

Результат обязательно преобразуется в ADR/decision log с проверяемыми утверждениями.

### Ограничения и красные зоны

- Запрет на отправку PII, secrets, клиентских NDA-данных в публичные LLM
- Верификация всех AI-генерированных решений человеком
- OWASP Top 10 for LLM Applications (2025) - в [threat model](artifacts.md#threat-model)
- Корпоративные настройки AI-ассистентов в IDE (что отправляется, как хранится)
- Регуляторные требования к AI - учитывать для клиентов в зарегулированных отраслях

См. также [AI-политика проекта](practices.md#9-ai-политика-проекта) | [Шаблон AI Policy](artifacts.md#ai-policy)

## API Design

### Contract-first подход

Сначала контракт (спецификация), потом реализация. API governance сдвигается "влево" - валидация на этапе дизайна.

**Стандарты:**
- **OpenAPI 3.1** - REST API (де-факто стандарт)
- **AsyncAPI** - event-driven контракты
- **CloudEvents** - стандарт CNCF для описания событий между брокерами/платформами

**Инструменты:**
- Линтер API-спецификаций (Spectral и аналоги) - валидация OpenAPI/AsyncAPI, встраивается в CI
- Платформа управления API lifecycle - коллаборативный дизайн, версионирование, публикация
- Инструмент тестирования API - ручное и автоматизированное тестирование контрактов

**Минимальный пакет на проекте:**
- OpenAPI/AsyncAPI спецификация в Git
- Style guide (набор правил линтера)
- ADR на ключевые API-решения

## Infrastructure as Code

IaC - "исполняемая часть архитектуры". Архитекторы формулируют политики, IaC их реализует.

**Policy-as-code:**
- **OPA (Open Policy Agent)** - унифицированный policy engine (CNCF graduated)
- **Conftest** - проверка конфигураций (Terraform, Kubernetes, Dockerfiles) против OPA-политик
- **Terraform Sentinel** - policy enforcement в Terraform Cloud/Enterprise

**Архитектурные политики (минимум):**
- Сетевые правила (сегментация, ingress/egress)
- Управление секретами (vault, не в коде)
- Шифрование (at rest, in transit)
- Теги стоимости (FinOps)
- Ограничения по регионам
- Drift detection - уведомления об отклонениях от целевого состояния
