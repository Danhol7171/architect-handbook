# Decision Rights & Governance Charter

## Назначение

Этот документ определяет **кто и как принимает архитектурные решения** на разных уровнях. Governance должен ускорять поток решений, а не тормозить его.

Принцип: **«решения принимаются на минимально достаточном уровне, эскалируются только при реальной необходимости»**.

---

## Классы решений

### Класс 1: Локальные решения (команда)

**Что:** решения внутри одного сервиса/модуля, не влияющие на контракты и другие команды.

**Примеры:**
- Выбор библиотеки / внутренней структуры модуля
- Рефакторинг без изменения контрактов
- Локальная оптимизация производительности
- Выбор подхода к тестированию компонента

**Кто принимает:** Tech Lead / [Technical Architect](roles.md#technical-architect-technology-architect) команды.
**Фиксация:** комментарий в PR или краткий ADR (опционально).
**SLA:** в рамках текущего спринта.

---

### Класс 2: Проектные решения (кросс-модульные)

**Что:** решения, затрагивающие несколько модулей, контракты между сервисами, NFR или архитектурный стиль.

**Примеры:**
- Добавление нового сервиса / изменение границ модулей
- Изменение API-контрактов (breaking changes)
- Выбор технологии для нового компонента (СУБД, message broker)
- Определение / изменение NFR targets
- Введение нового паттерна (CQRS, event sourcing, saga)

**Кто принимает:** [Solution Architect](roles.md#solution-architect) после design review.
**Фиксация:** обязательный [ADR](practices.md#1-architecture-decision-records-adr).
**Review:** асинхронно через PR или на [office hours](practices.md#2-архитектурное-ревью).
**SLA:** до 5 рабочих дней.

---

### Класс 3: Стратегические решения (кросс-проектные / портфельные)

**Что:** решения, влияющие на несколько проектов, enterprise standards или архитектурные принципы организации.

**Примеры:**
- Утверждение / изменение enterprise architecture standards
- Выбор enterprise-wide платформ (API Gateway, IDP, EAM)
- Изменение архитектурных принципов организации
- Решения о технологическом долге портфельного уровня
- Исключения из стандартов (waivers)

**Кто принимает:** Architecture Review Board (ARB).
**Фиксация:** ADR + протокол ARB.
**SLA:** до 15 рабочих дней (через очередное заседание ARB или внеочередное при срочности).

| Тип решения | Команда (Tech Lead / TA) | Solution Architect | ARB |
|-------------|:---:|:---:|:---:|
| Внутренняя реализация модуля | **Решает** | Информирован | — |
| Выбор библиотеки (без vendor lock-in) | **Решает** | Информирован | — |
| Новый сервис / изменение границ | Предлагает | **Решает** | Информирован |
| API breaking change | Предлагает | **Решает** | Информирован |
| Выбор СУБД / broker / cloud service | Предлагает | **Решает** | Информирован |
| Введение нового архитектурного паттерна | Предлагает | Рекомендует | **Решает** |
| Enterprise-wide стандарт | — | Предлагает | **Решает** |
| Исключение из стандарта (waiver) | — | Предлагает | **Решает** |
| Выбор enterprise-платформы | — | Предлагает | **Решает** |

## Механизм эскалации

### Когда эскалировать

1. Решение Класса 2, по которому нет консенсуса после 5 рабочих дней
2. Решение с высоким риском (security, значительный cost impact, регуляторика)
3. Решение, противоречащее принятым architecture standards
4. «Stop-the-line» ситуации (см. ниже)

### Путь эскалации

```
Tech Lead → Solution Architect → Lead Architect / ARB
```

На каждом уровне — попытка решить за 2 рабочих дня. Если не удалось — эскалация выше.

### Stop-the-line критерии

Немедленная эскалация на ARB (вне расписания):

- Обнаружена критическая уязвимость в архитектуре
- Решение ведёт к vendor lock-in с cost impact > 20% бюджета
- Нарушение регуляторных требований (ПДн, PCI DSS, ФЗ-152)
- Архитектурный drift > 30% от target architecture

## Architecture Review Board (ARB)

### Формат

- **Частота:** 1 раз в месяц (регулярное) + ad-hoc при stop-the-line
- **Длительность:** 60–90 минут
- **Участники:** Lead Architects, приглашённые SA по повестке

### Повестка (типовая)

1. Review решений Класса 3 (15 мин на решение)
2. Waiver requests — исключения из стандартов (10 мин на запрос)
3. Architecture standards updates (10 мин)
4. Метрики и conformance report (10 мин)
5. Open discussion (10 мин)

### Кворум

Минимум 3 Senior/Lead Architect. Решения принимаются консенсусом; при разногласии — голосование, Lead Architect имеет решающий голос.

### Результат

- ADR на каждое принятое решение
- Протокол заседания (краткий, в Git)
- Обновление Architecture Standards при необходимости

---

## Office Hours (проектный review)

### Формат

- **Частота:** еженедельно, 60–90 минут
- **Формат:** открытый слот, приходят архитекторы проектов с вопросами
- **Ведёт:** дежурный Lead/Senior Architect

### Что приносить

- ADR на review (draft)
- Архитектурные вопросы / дилеммы
- Запросы на design review сложных решений

### Результат

- Feedback на ADR (approve / request changes / escalate)
- Рекомендации по подходу
- Решение о необходимости эскалации на ARB

## SLA на решения

| Класс | SLA | Формат review |
|-------|-----|---------------|
| Класс 1 (локальное) | В рамках спринта | PR / самостоятельно |
| Класс 2 (проектное) | ≤ 5 рабочих дней | PR review + office hours |
| Класс 3 (стратегическое) | ≤ 15 рабочих дней | ARB meeting |
| Stop-the-line | ≤ 2 рабочих дня | Внеочередной ARB |

## Waivers (исключения из стандартов)

Иногда проект не может следовать стандарту. Это нормально, но должно быть явным.

### Процесс

1. SA оформляет [Waiver Request](../templates/waiver-request.md) с обоснованием отклонения
2. Указывает: какой стандарт, почему нельзя соблюсти, какие риски, план возврата к стандарту
3. Review на office hours → если одобряют, waiver действует
4. Если не одобряют → эскалация на ARB
5. Waiver имеет срок действия (не более 6 месяцев) и пересматривается

### Реестр waivers

Все действующие waivers фиксируются в едином [реестре waivers](../templates/waiver-register.md) (Git) и пересматриваются ежеквартально.
