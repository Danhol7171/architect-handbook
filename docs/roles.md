# Роли архитекторов

## Источники

Роли сформированы на основе лучших практик архитектурного консалтинга, дополнены современными компетенциями из исследования 2024-2026.

## Профили

### Solution Architect

**Фокус:** проектирование решения под бизнес-задачу, скоупинг, оценка, delivery model.

**Зона ответственности:**
- Формирование архитектуры решения от пресейла до сдачи
- Выбор компонентов и технологий (Build/Buy/Reuse)
- Управление архитектурными рисками
- Коммуникация с заказчиком по техническим вопросам
- Cost model как часть архитектуры (FinOps)
- Формирование и ведение ADR

**Ключевые артефакты:** [Solution Architecture (C4 L1-L2)](artifacts.md#solution-architecture-c4-based), [ADR](artifacts.md#architecture-decision-records-adr), [Fit/Gap Analysis](artifacts.md#fitgap-analysis), [NFR Checklist](artifacts.md#nfr-checklist)

**Участие в процессе:** ведущая роль на [пресейле](process.md#1-пресейл) и [инициации](process.md#2-инициация), ключевой участник [анализа](process.md#3-анализ-архитектуры) и [проектирования](process.md#4-проектирование-архитектуры)

**AI в работе:** генерация черновиков ADR и вариантов решений, анализ требований, подготовка trade-off analysis через LLM. См. [AI-инструменты](tools.md#ai-инструменты)

---

### Technical Architect (Technology Architect)

**Фокус:** техническая реализуемость, инфраструктура, NFR, интеграции, platform engineering.

**Зона ответственности:**
- Проектирование runtime, operations и infrastructure архитектуры
- Определение архитектурных требований (scalability, performance, security, availability)
- Менторинг команды разработки
- Архитектурное ревью кода и дизайна
- Определение fitness functions для автоматического контроля архитектуры
- Policy-as-code (OPA/Conftest) для инфраструктурных политик
- Zero Trust архитектура доступа

**Ключевые артефакты:** [Architecture Blueprints](artifacts.md#архитектурные-блюпринты), [Development Standards](artifacts.md#development-standards), [Fitness Functions](artifacts.md#fitness-functions), [Policy Rules](artifacts.md#policy-rules-policy-as-code)

**Участие в процессе:** ведущая роль на [реализации](process.md#5-реализация) и [тестировании](process.md#6-тестирование-архитектуры), ключевой участник [анализа](process.md#3-анализ-архитектуры) и [сопровождения](process.md#8-сопровождение)

**AI в работе:** терминальный AI-агент для reverse engineering кодовых баз, инструмент поведенческого анализа кода для оценки техдолга, AI-ассистент для навигации по незнакомому коду. См. [AI-инструменты](tools.md#ai-инструменты)

---

### Integration Architect

**Фокус:** межсистемное взаимодействие, потоки данных, API governance.

**Зона ответственности:**
- Проектирование интеграционных потоков
- Определение интерфейсных соглашений
- Выбор паттернов интеграции (sync/async, event-driven, batch)
- Contract-first подход: OpenAPI/AsyncAPI спецификации
- API governance и автоматический линтинг спецификаций

**Ключевые артефакты:** [Integration Design](artifacts.md#integration-design), Interface Agreement, [API Specifications](artifacts.md#api-specifications) (OpenAPI/AsyncAPI)

**Участие в процессе:** ведущая роль на [проектировании интеграций](process.md#4-проектирование-архитектуры), участие в [анализе](process.md#3-анализ-архитектуры)

**AI в работе:** генерация спецификаций из описания, анализ совместимости контрактов. См. [contract-first подход](practices.md#10-contract-first-api-design)

---

### Data Architect

**Фокус:** данные, их модели, миграция, качество, data products.

**Зона ответственности:**
- Моделирование данных
- Проектирование миграции и конвертации данных
- Data mapping между системами
- Data mesh / data fabric стратегия (на крупных трансформациях)
- Data contracts и качество данных

**Ключевые артефакты:** [Data Model / Data Definition](artifacts.md#data-model--data-definition), Data Mapping, Data Conversion Design, Data Product Specs

---

### Enterprise / Domain Architect (опционально)

> EA-lite — расширение для программ и трансформаций. Не является обязательной ролью для проектной работы. Подключается при [уровне rigor L3](core-standard.md#уровни-rigor).

**Фокус:** целевой ИТ-ландшафт, архитектурные стандарты организации, портфельные решения.

**Зона ответственности:**
- Capability mapping и target architecture
- Архитектурные стандарты организации (reference architectures)
- Портфельная рационализация (какие системы оставить, заменить, вывести)
- Координация архитектурных решений между проектами
- [ARB](governance-charter.md#architecture-review-board-arb) — стратегический уровень governance

**Ключевые артефакты:** Capability Map, Target Architecture Roadmap, Architecture Standards, Portfolio Rationalization Report

**Чем отличается от SA:**
- SA отвечает за архитектуру конкретного решения (project scope)
- EA отвечает за архитектуру ландшафта (organization scope)
- SA проектирует to-be для системы, EA — to-be для портфеля
- На практике один Senior SA может совмещать обе роли на небольших программах

**Границы:**
- Методика **не покрывает** полную стратегическую EA-функцию (CTO-офис, TOGAF enterprise continuum)
- EA в контексте методики = "EA-lite" — столько, сколько нужно для ведения трансформационных программ
- Для полного EA используется ArchiMate + EAM-платформа (см. [инструменты](tools.md#профили-инструментария))

## Как роли комбинируются на проектах

| Размер проекта | Типичный состав | Пример |
|---------------|----------------|--------|
| Малый (аудит, оценка) | 1 Solution Architect совмещает все роли | Аудит архитектуры, экспресс-оценка |
| Средний (внедрение модуля) | Solution Architect + Technical Architect | Разработка нового сервиса, модернизация |
| Крупный (трансформация) | SA + TA + Integration + Data + архитектурная команда | Миграция в облако, замена core-системы |
| Программа / портфель | + Enterprise Architect (опционально) | Трансформация ИТ-ландшафта, стандарты организации |

## Матрица компетенций

### Базовые компетенции (все профили)

| Компетенция | Junior | Middle | Senior | Lead |
|-------------|--------|--------|--------|------|
| C4 моделирование | Читает | Создаёт L1-L2 | Создаёт L1-L3, ревьюит | Определяет стандарты |
| ADR | Читает | Пишет с помощью | Пишет самостоятельно | Ревьюит, обучает |
| NFR | Знает категории | Собирает по чеклисту | Транслирует в решения | Определяет fitness functions |
| AI-инструменты | Использует для черновиков | Имеет prompt kit | Определяет AI-политику проекта | Формирует командные стандарты |
| Коммуникация | Документирует | Презентует команде | Презентует заказчику | Фасилитирует workshops |

### Дополнительные компетенции по профилям

**Solution Architect:**

| Компетенция | Что это на практике |
|-------------|-------------------|
| Облачная архитектура | Проектирование решений в облаке (Yandex Cloud, VK Cloud, Selectel и др.) с учётом отказоустойчивости, безопасности и стоимости |
| Управление стоимостью (FinOps) | Умение считать и обосновывать стоимость владения решением, оптимизировать расходы на инфраструктуру |
| Взаимодействие с бизнес-анализом | Работа на стыке с BA: трансляция бизнес-требований в архитектурные решения, участие в проработке бизнес-кейсов |

**Technical Architect:**

| Компетенция | Что это на практике |
|-------------|-------------------|
| Платформенная инженерия | Проектирование внутренних платформ разработки: портал разработчика (Backstage), шаблоны проектов, стандартные пути создания сервисов |
| Безопасность по модели нулевого доверия | Проектирование архитектуры доступа, сегментации сети и идентификации без "доверенного периметра" |
| Инфраструктура как код и политики | Описание инфраструктуры в Terraform/Pulumi, проверка соответствия политикам через OPA/Conftest |
| Наблюдаемость систем | Проектирование мониторинга, трассировки и логирования (OpenTelemetry, Grafana, ELK) |

**Integration Architect:**

| Компетенция | Что это на практике |
|-------------|-------------------|
| Проектирование API (contract-first) | Сначала спецификация (OpenAPI/AsyncAPI), потом реализация. Валидация и линтинг контрактов |
| Событийная архитектура | Проектирование асинхронных взаимодействий через брокеры сообщений (Kafka, RabbitMQ), стандарт CloudEvents |
| Интеграционные платформы | Опыт работы с ESB, iPaaS, API Gateway - выбор и настройка платформ интеграции |

**Data Architect:**

| Компетенция | Что это на практике |
|-------------|-------------------|
| Доменная организация данных (data mesh) | Организация работы с данными через доменные команды и "продукты данных" вместо централизованного хранилища |
| Контракты и качество данных | Описание формата, SLA и правил качества данных на границах между системами |
| Архитектура ML/AI-пайплайнов | Базовое понимание, как устроены конвейеры подготовки данных и обучения моделей |

## RACI по ключевым активностям

**Легенда:**
- **R** - Responsible (ответственный, делает работу)
- **A** - Accountable (утверждающий, принимает результат, один на активность)
- **C** - Consulted (консультируемый, даёт экспертизу)
- **I** - Informed (информируемый, получает результат для сведения)

| Активность | SA | TA | IA | DA |
|-----------|-----|-----|-----|-----|
| Концепция решения (пресейл) | R/A | C | C | - |
| Архитектурные принципы | R/A | C | I | I |
| Архитектура решения (C4) | R/A | R | C | C |
| Нефункциональные требования | A | R | C | C |
| Архитектурные блюпринты | A | R/A | R | R |
| Проектирование интеграций | C | C | R/A | C |
| Архитектура данных | C | C | C | R/A |
| Архитектурные решения (ADR) | R | R | R | R |
| Архитектурное ревью | R/A | R | R | R |
| Автоматические проверки (fitness functions) | A | R/A | C | C |
| Управление техдолгом | C | R/A | C | C |
